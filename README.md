# Maker Space MVP

Веб-приложение для внутренней работы университетского Maker Space: учёт пользователей, допуск по инструктажу, каталог оборудования, бронирование станков, подтверждение заявок персоналом, уведомления и журнал действий.

Репозиторий на GitHub: [MishOchEK-1/MSproject](https://github.com/MishOchEK-1/MSproject)

## Что это за проект

Проект решает типовой сценарий мастерской или учебного цеха:

- пользователь регистрируется и входит в систему по `email`;
- видит доступное оборудование и ограничения по допуску;
- создаёт заявку на бронь;
- персонал подтверждает или отклоняет заявку;
- система показывает общую загрузку оборудования по дням и часам;
- все ключевые действия логируются в аудит и сопровождаются внутренними уведомлениями.

Это серверный Django-проект без REST API: интерфейс построен на Django templates, CBV и стандартной admin-панели.

## Основные возможности

- кастомная модель пользователя с ролями `guest`, `student`, `staff`, `admin`;
- вход по `email`, а не по `username`;
- каталог оборудования с категориями, фото, фильтрами и карточками;
- расписание загрузки `/equipment/schedule/` по всему оборудованию;
- индивидуальная дневная шкала в карточке оборудования;
- бронирование с шагом `20` минут;
- проверка пересечений броней и периодов недоступности;
- ручное подтверждение заявок персоналом;
- автоматическое подтверждение для `staff` и `admin`;
- продление и отмена активных броней;
- статусы брони: `pending`, `approved`, `rejected`, `cancelled`, `completed`, `expired`;
- внутренний центр уведомлений;
- журнал аудита по действиям пользователей и системы;
- экран управления инструктажем;
- готовые миграции и демо-наполнение для локального показа проекта.

## Роли и доступ

| Роль | Что может |
|---|---|
| `guest` | Просмотр доступного оборудования, бронирование только там, где не нужен инструктаж |
| `student` | Бронирование после прохождения инструктажа, просмотр своих броней и уведомлений |
| `staff` | Автоподтверждение своих броней, проверка заявок, управление инструктажем, просмотр аудита |
| `admin` | Все права `staff` + административный контур Django |

Дополнительно в проекте есть бизнес-правило: для ролей `student` и `staff` требуется корпоративная почта `@edu.omsk.ru`.

## Бизнес-правила бронирования

- минимальная длительность брони: `20` минут;
- максимальная длительность брони: `24` часа;
- шаг времени: `20` минут;
- бронь нельзя создать дальше чем на `3` недели вперёд;
- пересечение активных броней на одном оборудовании запрещено;
- бронирование недоступно, если оборудование не в статусе `available`;
- бронирование недоступно на периодах downtime;
- заявки обычных пользователей уходят в `pending`;
- заявки `staff` и `admin` подтверждаются автоматически;
- неподтверждённые заявки истекают через `7` дней;
- завершённые подтверждённые брони переводятся в `completed` автоматически.

## Модули проекта

- `users` — регистрация, логин, профиль, роли, инструктаж;
- `equipment` — категории, карточки оборудования, загрузка, downtime;
- `reservations` — создание, подтверждение, отклонение, отмена, продление;
- `notifications` — внутренние уведомления по событиям бронирования;
- `audit` — журнал действий для сотрудников и администраторов;
- `main` — главная страница и базовые сущности проекта.

## Быстрый старт

### 1. Клонирование

```bash
git clone https://github.com/MishOchEK-1/MSproject.git
cd MSproject
```

### 2. Виртуальное окружение

```bash
python3 -m venv .venv
source .venv/bin/activate
```

### 3. Установка зависимостей

```bash
pip install -r requirements.txt
```

### 4. Миграции

```bash
python manage.py migrate
```

### 5. Создание суперпользователя

```bash
python manage.py createsuperuser
```

### 6. Запуск сервера

```bash
python manage.py runserver
```

После запуска приложение доступно по адресам:

- главная: `http://127.0.0.1:8000/`
- каталог оборудования: `http://127.0.0.1:8000/equipment/`
- общая загрузка: `http://127.0.0.1:8000/equipment/schedule/`
- мои брони: `http://127.0.0.1:8000/reservations/`
- уведомления: `http://127.0.0.1:8000/notifications/`
- аудит: `http://127.0.0.1:8000/audit/`
- админка: `http://127.0.0.1:8000/admin/`

## Демо-данные

После миграций проект уже содержит стартовое оборудование через миграции приложения `equipment`. Для живой демонстрации можно заполнить расписание тестовой активностью:

```bash
python manage.py seed_demo_activity --days 12
```

Команда создаёт:

- демо-пользователей разных ролей;
- прошлые завершённые брони;
- будущие подтверждённые брони;
- ожидающие подтверждения заявки.

Пароль для демо-аккаунтов:

```text
demo-pass-123
```

Примеры демо-пользователей:

- `demo_staff` / `demo.staff@edu.omsk.ru`
- `demo_student_1` / `demo.student1@edu.omsk.ru`
- `demo_student_2` / `demo.student2@edu.omsk.ru`
- `demo_guest_1` / `guest.demo1@example.com`

Важно: вход в интерфейс выполняется по `email`.

## Что особенно важно в реализации

- права доступа вынесены в роли и проверки на уровне моделей и view;
- жизненный цикл брони синхронизируется сервисным слоем;
- подтверждение, отмена и продление выполняются через транзакционные сервисы;
- действия пользователей и системы логируются в `audit`;
- уведомления создаются на событиях бронирования;
- расписание учитывает и брони, и периоды технической недоступности;
- в проекте есть покрытие тестами ключевых пользовательских сценариев.

## Ограничения текущего MVP

- используется SQLite, а не production-ready СУБД;
- нет REST API и внешней интеграции;
- нет Docker-конфигурации;
- нет фоновых задач и очередей;
- нет CI/CD-конвейера в репозитории;
- нет отдельного production-конфига с переменными окружения.

## Что можно развивать дальше

- вынести настройки в `.env`;
- добавить PostgreSQL и Docker Compose;
- подключить CI через GitHub Actions;
- добавить экспорт расписания и отчётов;
- реализовать REST API или DRF;
- добавить календарный вид по неделям и месяцам;
- покрыть проект e2e-тестами.
